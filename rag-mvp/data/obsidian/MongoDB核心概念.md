学习 [[MongoDB]] 不仅仅是学习如何存储 JSON，更重要的是理解其**非关系型**的设计哲学。如果你能掌握索引、聚合和建模，你就已经跨过了 MongoDB 从“入门”到“精通”的门槛。

---

## 1. [[索引]] (Indexes)：查询性能的推进器

没有索引的 MongoDB 就像一本书没有目录。当你查询时，数据库必须进行 **COLLSCAN（全表扫描）**，这在数据量大时是灾难性的。

### 核心要点

- **复合索引 (Compound Index)**：`db.collection.createIndex({ age: 1, name: -1 })`。注意**字段顺序**很重要（最左前缀原则）。
    
- **覆盖查询 (Covered Query)**：如果索引包含查询所需的所有字段，MongoDB 甚至不需要读取文档本身，直接从索引返回结果，速度极快。
    
- **唯一索引 (Unique Index)**：确保集合中某个字段的值是不重复的。
    
- **TTL 索引**：用于自动删除过期的文档（如验证码、临时日志）。
    

> [!TIP]
> 
> **优化金律**：使用 `.explain("executionStats")` 来观察你的查询。如果 `totalDocsExamined` 远大于 `nReturned`，说明你的索引需要优化。

## 2. [[聚合管道]] (Aggregation Pipeline)：复杂数据的处理工厂

聚合管道类似于 Linux 的管道命令（`|`），它将文档传入一个由多个阶段（Stages）组成的流水线，每个阶段对数据进行转换。

### 常用阶段示例

JavaScript

```
db.orders.aggregate([
    { $match: { status: "urgent" } },                // 1. 筛选：只看紧急订单
    { $group: { _id: "$product", total: { $sum: "$amount" } } }, // 2. 分组：按产品计算总额
    { $sort: { total: -1 } },                        // 3. 排序：按总额降序
    { $limit: 5 }                                   // 4. 截取：前 5 名
]);
```

- **$lookup**：实现类似于 SQL 的 `JOIN` 功能（多表关联）。
    
- **$project**：重构文档结构，只保留需要的字段或添加计算字段。

## 3. 数据关系设计：[[嵌入 (Embedding) vs 引用 (Referencing)]]

这是 MongoDB 建模中最核心的决策。没有标准答案，只有“最适合应用场景”的答案。

|**特性**|**嵌入 (Embedding)**|**引用 (Referencing)**|
|---|---|---|
|**定义**|数据直接以嵌套文档形式存放在一起|存储另一个文档的 `_id`|
|**关系类型**|一对一，或“一对很少”|一对多，多对多|
|**查询性能**|**极快**（一次查询获取所有数据）|**较慢**（需要多次查询或使用 `$lookup`）|
|**一致性**|强（单个文档原子性更新）|弱（需要处理孤儿数据或手动同步）|
|**限制**|文档最大 16MB 限制|灵活，不受大小限制|

## 4. Schema 设计最佳实践

虽然 MongoDB 是 "Schema-less" 的，但在生产环境下，**“无模式”不代表“无设计”**。

1. **优先考虑读模式**：MongoDB 是为应用服务的。根据你的前端页面显示什么数据来决定如何存储，而不是根据传统的范式。
    
2. **避免巨大的数组**：如果一个文档的数组字段会无限增长（如：成千上万条评论），**千万不要嵌入**，请改用引用。
    
3. **避免深层嵌套**：嵌套层级过深（超过 3 层）会让查询和更新变得异常痛苦。
    
4. **一致性字段名**：虽然 MongoDB 不强制，但请务必保持字段命名一致（如不要有的地方叫 `userId`，有的地方叫 `u_id`）。
    
5. **数据冗余有理**：为了减少 `$lookup`，在嵌入文档中冗余存储一些常用的不常变动的字段（如用户头像、姓名）是完全可以接受的。
    

---

## 5. 总结

- **索引**：消除全表扫描，利用 `explain` 调试。
    
- **聚合**：像搭积木一样处理复杂逻辑。
    
- **建模**：能嵌入就嵌入，只有数据量大到会突破 16MB 或需要频繁独立维护时才考虑引用。
