# 一、需求文档 PRD

## 1.1 项目背景

本项目旨在通过构建一个轻量级 **基于大模型的 Advanced RAG（高级检索增强生成）** 知识库系统，采用 **向量检索（Dense Retrieval）** 结合大模型的 **生成式问答（Generative QA）** 技术，帮助开发者深刻理解并掌握**大模型时代的 RAG 核心技术链路**，区别于传统的关键词检索和抽取式系统，为后续构建带智能体特性的知识管理系统（Agentic RAG）奠定坚实的技术基础。

## 1.2 项目目标

- **主要目标**：搭建一个可运行的端到端 RAG 系统，理解并掌握 Embedding、向量检索、上下文注入等核心概念
- **次要目标**：为后续对接 Obsidian 知识库积累技术经验

## 1.3 用户角色

本项目只有一个用户角色：**开发者本人**（个人学习使用）

## 1.4 功能需求

### 核心功能

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F-01 | 文档导入 | 支持上传本地 txt / md 文件，解析并存入知识库 | P0 |
| F-02 | 文本分块 | 将长文档自动切分为合适大小的文本块（chunk） | P0 |
| F-03 | 向量化存储 | 对每个文本块生成 Embedding 向量并持久化 | P0 |
| F-04 | 向量语义检索 (Dense Retrieval) | 将问题向量化，通过计算稠密向量相似度检索最相关的 Top-K 文本块，而非传统关键词匹配 | P0 |
| F-05 | 生成式问答 (Generative QA) | 将检索结果作为上下文注入 LLM Generator（生成器），由大模型归纳重写生成最终回答 | P0 |
| F-06 | 命令行交互 | 通过命令行进行问答交互 | P1 |
| F-07 | 简单 Web UI | 提供基础的浏览器交互界面 | P2 |

### 非核心功能（本项目不包含）

- 用户登录/权限管理
- 多用户隔离
- 生产级性能优化
- PDF / Word 文档解析

## 1.5 非功能需求

- **简单性**：代码结构清晰，适合学习，优先可读性而非性能
- **可调试性**：每个环节能输出中间结果（分块内容、检索到的片段、最终 Prompt）
- **本地运行**：所有组件可在本地运行，无需服务器部署
- **成本可控**：LLM 调用可切换为本地模型，降低 API 费用

## 1.6 约束条件

- 开发语言：Python 3.10+
- 运行环境：本地 Mac / Windows / Linux
- 预算：API 费用控制在学习阶段可接受范围内
- 时间：4 周完成

## 2.3 核心模块设计

### 模块一：文档加载与分块

**职责**：读取文档，切成合适的文本块（chunk）

```
文档文件（txt/md）
      ↓
 DocumentLoader（读取原文）
      ↓
 TextSplitter（切分策略）
      ↓
 List[Chunk]（文本块列表）
```

**关键参数**：
- `chunk_size`：每块字符数，建议 **500~1000**（中文）
- `chunk_overlap`：块之间重叠字符数，建议 **50~100**，防止语义截断

**分块策略**（按优先级）：
1. 先按段落（`\n\n`）切分
2. 段落仍然太长，按句子切分
3. 句子仍然太长，按字符数硬切

### 模块二：向量化与存储

**职责**：将文本块转为向量并存入 ChromaDB

```
List[Chunk]
     ↓
 EmbeddingModel.encode()
     ↓
 List[Vector]（浮点数组）
     ↓
 ChromaDB.add()（存储向量 + 原始文本 + 元数据）
```

**ChromaDB 集合结构**：
```python
{
  "id": "chunk_001",
  "embedding": [0.12, -0.34, ...],   # 1536 维向量
  "document": "原始文本内容",
  "metadata": {
    "source": "文件名.md",
    "chunk_index": 0,
    "created_at": "2026-02-13"
  }
}
```

### 模块三：向量检索 (Dense Retrieval)

**职责**：区别于传统基于关键词匹配的稀疏检索（Sparse Retrieval），将用户问题转化为稠密向量，在向量空间中寻找语义上最相近的文本块

```
用户问题（文本）
     ↓
 EmbeddingModel.encode()
     ↓
 ChromaDB.query(n_results=5)    ← Top-K 相似度检索
     ↓
 List[相关文本块]（按相似度排序）
```

**检索策略**：
- 默认使用**余弦相似度**（ChromaDB 默认支持）
- Top-K 建议取 **3~5**，过多会稀释上下文质量

### 模块四：Prompt 组装与大模型生成 (Generative QA)

**职责**：区别于传统仅靠抽取原文片段的抽取式问答（Extractive QA），本模块负责将零散的检索结果组装为 Prompt 提供给 LLM Generator，让大模型分析并用自然语言重新生成回答

**Prompt 模板**：
```
你是一个知识库助手，请根据以下参考资料回答用户问题。
如果参考资料中没有相关信息，请直接说"知识库中未找到相关内容"，不要编造答案。

【参考资料】
{context}

【用户问题】
{question}

【回答】
```

# 三、验收标准

## 3.1 功能验收

### F-01 文档导入

- [ ] 能成功读取 `data/` 目录下的 `.txt` 和 `.md` 文件
- [ ] 读取失败时有明确的错误提示
- [ ] 支持一次导入单个文件和批量导入整个目录

### F-02 文本分块

- [ ] 能将 1000 字以上的文档正确切分为多个文本块
- [ ] 每块大小在设定的 `chunk_size` ± 20% 范围内
- [ ] 相邻块之间存在 `chunk_overlap` 字符的重叠
- [ ] **可调试**：能打印输出所有分块的内容和数量

### F-03 向量化存储

- [ ] 文档导入后，向量数据能持久化保存（重启程序后仍存在）
- [ ] 相同文档重复导入时不产生重复向量（去重机制）
- [ ] **可调试**：能查询当前知识库中共有多少个文本块

### F-04 语义检索

- [ ] 输入问题后，能返回 Top-3 到 Top-5 个相关文本块
- [ ] 检索结果与问题语义相关（人工判断合理性）
- [ ] 能展示每个检索结果的相似度分数
- [ ] **可调试**：能单独运行检索，查看原始检索结果

### F-05 问答生成

- [ ] 最终回答基于检索到的文本块内容，而非 LLM 训练知识
- [ ] 知识库中没有相关内容时，回答"未找到"而非胡编
- [ ] 回答中能体现出使用了哪个来源文件的内容

### F-06 命令行交互

- [ ] 执行 `python cli.py --ingest data/` 能完成知识库构建
- [ ] 执行 `python cli.py --query "你的问题"` 能返回回答
- [ ] 执行 `python cli.py --chat` 能进入多轮对话模式

### F-07 Web UI（P2，可选）

- [ ] 执行 `streamlit run app.py` 能打开浏览器界面
- [ ] 界面包含：文件上传区、问题输入框、回答展示区
- [ ] 展示检索到的原文片段（来源溯源）

## 3.2 可调试性验收（学习目标专项）

这是练手项目的核心，每个环节都要能"看到"中间过程：

| 调试点 | 验收内容 |
|--------|---------|
| 分块可视化 | 能打印每个 chunk 的编号、字数、内容预览 |
| 向量可视化 | 能打印某个 chunk 对应向量的维度和前 5 个值 |
| 检索可视化 | 能打印 Top-K 检索结果、相似度分数、来源文件 |
| Prompt 可视化 | 能打印最终发送给 LLM 的完整 Prompt |
| 对比验证 | 用同一个问题分别"有知识库"和"无知识库"问 LLM，对比回答差异 |

## 3.3 代码质量验收

- [ ] 每个模块（loader / embedder / retriever / generator）可独立运行和测试
- [ ] `README.md` 包含完整的安装和运行说明
- [ ] 关键函数有注释说明参数含义
- [ ] 敏感信息（API Key）通过 `.env` 管理，不硬编码在代码中

## 3.4 学习目标验收（主观）

完成项目后，开发者应能够口头解释以下问题：

- [ ] LLM-based RAG 这一类生成式问答（Generative QA）与传统的抽取式问答（Extractive QA）的本质区别在哪？
- [ ] 我们系统所用的向量检索（Dense Retrieval）相比于传统的关键词检索（Sparse Retrieval，如 BM25），其优势和劣势分别是什么？
- [ ] RAG 和直接问 LLM 的本质区别是什么？
- [ ] chunk_size 太大或太小分别有什么问题？
- [ ] 向量相似度是如何衡量"语义相近"的？
- [ ] Top-K 的 K 值如何影响回答质量？
- [ ] 为什么需要 chunk_overlap？
