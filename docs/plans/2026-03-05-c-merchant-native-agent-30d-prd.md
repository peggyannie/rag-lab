# 面向通用型小商户的原生 Agent（基于 rag-mvp）30 天实施版 PRD

> 日期：2026-03-05  
> 适用范围：`rag-mvp` 增量演进  
> 优先方案：客服成交 Agent（先落地），兼容后续扩展到获客增长 Agent
> 文档定位：战略与范围基线（非 AI Agent 一线执行手册）
> 执行入口：请优先使用 `docs/agent/`（尤其 `02-task-backlog.md` 与 `03-task-card-template.md`）

## 1. 背景与目标

当前 `rag-mvp` 已具备以下可复用能力：
1. 文档导入与切块（`.md/.txt`）。
2. 检索增强问答（`local` / `openai` 双 Provider）。
3. 来源展示与调试视图（Streamlit + CLI）。

30 天目标：
1. 从“通用 RAG Demo”升级到“可服务小商户咨询转化”的可用 Agent。
2. 支持高频经营咨询自动应答、成交引导、人工接管。
3. 在不重构主架构的前提下，先完成单渠道闭环 MVP。

## 2. 用户与场景定义

目标用户：通用型 C 端商户小老板（餐饮、零售、个体服务业通用）。

高频场景：
1. 顾客咨询商品/价格/配送/售后。
2. 顾客犹豫下单，需推荐方案和促单话术。
3. 夜间无人值守，需自动回复并保留线索。

## 3. 产品范围

P0（必须，30 天内上线）：
1. 商户知识库模板化导入（商品、配送、售后、门店信息）。
2. 高频问答自动回复（Top 30 问题）。
3. 引用来源返回（减少幻觉，便于老板核验）。
4. 人工接管开关（高风险问题不自动答）。
5. 会话日志与基础指标（响应时长、命中率、转人工率）。

P1（建议，30 天内并行完成 30%~60%）：
1. 成交引导策略（搭配推荐、优惠提醒、催付话术）。
2. 多轮上下文（单会话短记忆，最近 N 轮）。
3. 运营后台最小页（知识条目管理、命中问题榜单）。

P2（延后，30 天后）：
1. 多渠道接入（平台 IM + 私域统一收口）。
2. 混合检索升级（BM25 + 向量）。
3. 精排模型与自动评测（Ragas/离线评估集）。

非目标（本期不做）：
1. 全行业深度定制流程引擎。
2. 复杂工单系统与 CRM 全量替代。
3. 强依赖外部平台官方 API 的深度自动化履约。

## 4. 方案设计（基于现有 rag-mvp 增量）

架构增量：
1. 保留 `src/pipeline.py` 作为检索与生成核心。
2. 新增 `agent` 层：会话管理、风控路由、成交策略。
3. 新增轻量 API 层（建议 FastAPI）：给 Web/渠道机器人统一调用。

关键模块：
1. Knowledge Builder：商户资料模板导入与清洗。
2. Agent Orchestrator：问题分类（问答/风险/促单）与策略分流。
3. Guardrail：风险词和超范围问题拦截，强制转人工。
4. Conversation Store：保存 session_id、最近对话、状态标记。
5. Analytics：日志埋点与运营看板基础数据。

回答策略：
1. 先检索，后生成；无足够证据时输出“需人工确认”。
2. 每条回答附来源片段与置信提示。
3. 促单策略仅在“非投诉、非售后争议”场景启用。

## 5. 30 天执行计划（P0/P1/P2）

第 1 周（Day 1-7）- P0 基础搭建：
1. 定义商户知识库标准模板（商品、门店、配送、售后）。
2. 完成导入校验与字段规范化。
3. 建立 FAQ 基线集（30-50 条）与测试问集。
4. 输出首版回答协议（answer + sources + risk_flag + handoff）。

验收：
1. 能稳定 ingest 模板数据并查询。
2. FAQ 问题命中率达到可测状态（有基线报告）。

第 2 周（Day 8-14）- P0 核心闭环：
1. 增加 Agent Orchestrator（问答/风险/转人工路由）。
2. 接入人工接管开关与兜底话术。
3. 打通 API（至少 `/chat`、`/handoff`、`/health`）。
4. 增加会话日志存储（最小字段集）。

验收：
1. 高风险问题 100% 不自动编造答案。
2. 单轮咨询链路可跑通：提问 -> 回答/转人工 -> 记录。

第 3 周（Day 15-21）- P1 能力增强：
1. 增加短会话记忆（最近 N 轮上下文注入）。
2. 增加成交引导策略库（优惠、搭配、催付）。
3. 建立运营统计页最小集（Top 问题、转人工率、响应时长）。
4. 迭代 prompt 与风险规则。

验收：
1. 多轮追问可保持上下文一致性。
2. 促单场景较基线有正向提升（离线对比）。

第 4 周（Day 22-30）- P0 上线与 P2 预研：
1. 进行灰度上线（1-3 家试点商户）。
2. 建立线上告警（错误率、超时率、空回答率）。
3. 完成复盘并形成 P2 需求包（多渠道、混合检索、评测体系）。
4. 固化运维手册与回滚策略。

验收：
1. 可稳定服务试点商户连续 7 天。
2. 输出下阶段排期与技术债清单。

## 6. KPI 与验收口径

业务 KPI：
1. 首响时长下降 >= 50%。
2. 自动应答覆盖率 >= 60%。
3. 转人工率控制在 20%~40%（按行业可调）。
4. 咨询到下单转化率提升 >= 8%（试点口径）。

技术 KPI：
1. P95 响应时间 <= 3.5s（不含外部平台延迟）。
2. 空回答率 <= 3%。
3. 高风险误答率 <= 1%。
4. 服务可用性 >= 99%。

## 7. 风险与缓解

主要风险：
1. 商户资料质量差，导致检索噪声高。
2. 促单话术在投诉场景误触发。
3. 单一向量检索在关键词问题上召回不足。

缓解策略：
1. 上线前强制模板校验和缺失项提示。
2. 风险分类优先级高于促单策略。
3. P2 引入 BM25 混合召回并建立离线评测集。

## 8. 资源与依赖

人员建议（最小编制）：
1. 后端/算法 1 人。
2. 前端 1 人（可兼职）。
3. 产品/运营 1 人（定义话术与评测口径）。

外部依赖：
1. LLM API 配额与成本预算。
2. 试点商户真实问答样本。
3. 可落地的人工接管通道（企业微信/飞书/店主端）。

## 9. 版本结论

本 PRD 结论：
1. 30 天内优先做“客服成交 Agent”是对当前 `rag-mvp` 复用率最高、见效最快的路径。
2. 获客增长 Agent 可在同一知识底座上并行孵化，但建议作为 P1/P2 延伸。
3. 经营管家 Agent 需先补结构化经营数据层，不建议抢在本期主线前。
